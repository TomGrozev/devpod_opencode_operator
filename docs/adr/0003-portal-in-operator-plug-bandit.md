# Serve the devpod portal in-operator via Plug + Bandit

The web portal that lists current devpod workspaces and their links lives inside the operator Elixir application, served by Bandit over a Plug router, with HTML rendered via stdlib `EEx`. Phoenix is not a dependency.

**Considered options:**

- **Co-located in-operator via Plug + Bandit (chosen).** The portal's whole job is to surface what the operator already knows — workspaces and their `https://<id>.<base_domain>` links — so co-locating lets the route reuse `DevpodOpencodeOperator.K8s` (the `list_pods` facade and its Mox-backed test seam), the `Workspace.from_pod/2` derivation, and `Config` (which already holds `base_domain` and `target_namespace`). Bandit is a pure-Elixir HTTP server with zero native deps; Plug + Bandit is a ~5-line supervision child and a ~20-line router. One HTML page does not justify Phoenix's endpoint config, PubSub, template tree, or `*Web` namespace convention. Consistent with ADR-0002's stated tone of "direct, not framework-y."

- **Full Phoenix (rejected).** Overkill for a single static-ish page; pulls in `Phoenix.PubSub`, `Phoenix.Template`, endpoint plumbing, and a `*Web` namespace that doesn't match a K8s controller repo. The dependency surface grows disproportionately to the feature. Reach for it only if the portal grows real interactivity (LiveView, logs, restart buttons, multi-namespace views).

- **A separate app/rep (rejected).** Duplicates K8s connection wiring and the `Workspace` derivation logic.

- **A static page regenerated by an external job (rejected).** Stale; the cluster is already right there.

- **Served by the gateway itself (rejected).** The gateway routes workspace traffic to OpenCode; conflating concerns.

**HTML rendering:** stdlib `EEx` (no new deps). `EEx` wins over HEEx because HEEx requires a `phoenix_live_view` dep, and a single static table of workspace links doesn't need syntax-checked component composition. Migrating from a Plug app to Phoenix later is a well-trodden path; the reverse is painful, so picking the smaller thing first is the lower-regret direction.

**Consequences:** the operator's supervision tree gains a `Bandit` child serving a Plug router under `DevpodOpencodeOperator.Portal`. The portal routes reuse `K8s.list_pods/2` against the existing `:k8s_cluster` facade, so the same Mox-backed test seam covers portal rendering tests. The portal is read-only and shares the operator's cluster credentials — no separate auth layer is introduced yet.
